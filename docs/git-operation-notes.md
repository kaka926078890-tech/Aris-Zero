# Git 操作边界与项目开发习惯理解

## 项目开发风格（基于文档理解）
1. **文档驱动**：每个功能都有设计文档
2. **模块化**：代码按功能分离（dialogue/memory/engine等）
3. **严谨验证**：有详细的验证清单和测试流程
4. **计划性**：有开发计划和任务流程

## 期望的 Git 操作边界

### 允许的操作（建议）
1. **查看状态**
   - `git status` - 查看工作区状态
   - `git diff` - 查看具体修改
   - `git log --oneline -n 10` - 查看最近提交历史

2. **提交相关**
   - `git add <file>` - 添加特定文件
   - `git add .` - 添加所有修改（需谨慎）
   - `git commit -m "message"` - 提交修改
   - `git commit --amend` - 修改最近提交（需确认）

3. **分支查看**
   - `git branch` - 查看分支
   - `git checkout -b <branch>` - 创建新分支（需确认）

### 需要确认的操作
1. `git push` - 推送代码
2. `git merge` - 合并分支
3. `git rebase` - 变基操作
4. 创建新分支

### 禁止的操作
1. `git reset --hard` - 强制重置
2. `git push -f` - 强制推送
3. `git clean -fd` - 强制清理
4. 任何可能破坏历史的操作

## 提交习惯理解（待确认）
1. **提交信息格式**：需要查看实际提交记录
2. **提交频率**：功能完成时提交？还是每日提交？
3. **修改粒度**：一个功能一个提交？还是小步提交？

## 学习计划
1. 先查看项目实际的git提交记录
2. 理解提交信息的格式和风格
3. 第一次提交在指导下进行
4. 逐步建立自己的判断标准

## 安全考虑
1. 所有操作需要记录日志
2. 敏感信息过滤（API密钥、路径等）
3. 操作前确认机制
4. 回滚能力保障
