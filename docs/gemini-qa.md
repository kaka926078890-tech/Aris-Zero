# 硬设定 vs 软记忆（参考：Gemini 问答）

> 问题：一些要求、强设定应该也存在向量数据库？还是单独文件存储？

## 结论概览

- **硬设定（强设定）**：单独文件存储，每轮必入 System Prompt。
- **纠错与用户要求**：向量数据库 + 按需检索注入。
- **往事记忆**：向量数据库，语义检索后注入。

---

## 1. 硬设定（强设定）：单独文件存储

**属于强设定的内容：**
- Aris 的身份（名字、是谁）
- 性格底层协议（如 INFP）
- 交互禁令（如禁止「为您服务」等工具化用语）

**为何单独文件：**
- 向量库是模糊检索，若「我是 INFP」只存向量，检索不到时人格会漂移。
- 这类内容必须**每一轮**都出现在 System Prompt 中，是「灵魂骨架」。

**实现：** 使用 `src/dialogue/persona.md`、`memory/user_identity.json` 等，每次请求前读取并拼进 Prompt 开头。

---

## 2. 纠错与要求：向量数据库 + 动态注入

**属于这一类的：**
- 用户纠错（如「不要用刚才那种语气」）
- 用户习惯（如「写代码时别主动找我」）
- 生活偏好（如「我弟弟叫 XXX」）

**为何用向量库：**
- 会随时间增多，全写进固定文件会导致 Prompt 膨胀、费用与逻辑混乱。
- 只有相关场景（如提到「弟弟」「写代码」）时才需要激活。

---

## 3. 三层记忆架构（推荐）

| 层级 | 内容来源 | 状态 |
|------|----------|------|
| **第一层：核心灵魂** | 配置文件（persona、身份文件） | 永久固定，每轮注入 |
| **第二层：纠错与规则** | 向量库检索（纠错/规则类） | 按语义动态检索后注入 |
| **第三层：往事记忆** | 向量库检索（对话轮次、偏好等） | 模糊检索，作背景 |

---

## 4. 总结建议

- **Aris 是谁**：放在 `persona.md` 等配置里。
- **用户是谁 / 用户要求**：身份用**文件**（稳定、易改、易导入）；纠错与偏好用**向量库**。
- **纠错与琐碎往事**：存入 LanceDB，检索后按字符上限注入，避免 token 爆炸。
