# 回复内容被截断的原因排查（2026-03）

## 现象

- 助手长回复（如自我优化总结）在固定位置被截断，例如「实现了情感标签自动提取,从我的」后面就没有了。
- 重试或让模型「重新完整输出」仍会在同一位置截断。

## 原因结论

**流式对话接口 `chatStream` 使用了 `max_tokens: 1024`，API 在生成满 1024 个 token 后停止，导致长回复被服务端截断。**

- **位置**：`src/dialogue/api.js` 第 6 行 `MAX_TOKENS_STREAM = 1024`，第 125 行 `chatStream` 请求里使用 `max_tokens: MAX_TOKENS_STREAM`。
- **影响**：无论「无 tool 直接流式」还是「有 tool 后第二次调用流式」，最终回复都走 `chatStream`，因此都受 1024 token 限制。中文约 1～2 字/token，即约 500～1000 字后就可能被截断，截断点会随内容长度稳定出现在相近位置。
- **为何同一处截断**：同一类长回复（如自我优化总结）长度相近，达到 1024 token 的位置相近，所以看起来总是在「从我的」附近断掉。

## 未截断的环节（已排除）

- **前端**：无按长度截断；`filterReplyForDisplay` 只做标签/情感摘要过滤，不限制长度；`sendReplyAsChunks` 按 2 字一块发送，会发完 `contentForFrontend` 全文。
- **会话落库**：`append(sessionId, 'assistant', reply)` 存的是 handler 拿到的 `reply`，而 `reply` 本身已是 API 返回的截断后内容，故数据库里也是截断后的。
- **IPC**：主进程 `sendChunk(chunk)` 仅发送小片段，无长度限制问题。

## 当初为何设置 max_tokens？

代码里没有注释说明，从常见做法可以推断：

1. **控制成本**：计费按 token（输入+输出），设上限可以避免单次回复暴走、费用不可控。
2. **控制延迟**：流式会在达到 max_tokens 时结束；设得很大时，若模型「话痨」，要等很久才结束。
3. **保守默认**：1024 对「日常短回复」够用（几百字），当时可能没考虑长总结、自我优化汇报这类场景。

所以 1024 更像是一个**偏保守的默认值**，而不是有特殊业务原因。

## 不设置会怎样？

- **必须传 max_tokens 的 API**：不传可能报错或拒绝请求（视厂商文档而定）。
- **有默认值的 API**：不传则用厂商默认值。DeepSeek 等常见默认多为 4096～8192，即**不设的话，往往反而能输出更长**，不会在 1024 就断。
- **风险**：不设或设得很大，单次回复可能特别长，导致费用略增、流式时间变长；一般用 4096/8192 对「长总结」足够且可控。

## 修改建议（供后续实现）

- 将流式回复的 `max_tokens` 提高，例如改为与 tool 轮一致或单独环境变量（如 `MAX_TOKENS_STREAM` 改为 4096 或 8192），或通过 `process.env.ARIS_STREAM_MAX_TOKENS` 配置，避免长总结再次被截断。
